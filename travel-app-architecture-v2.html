02

Two Distinct Modes

🗓 Planning Mode

User hasn't left yet. They input destination, dates, hotel (optional). AI generates full itinerary using Reddit + Google data. User can drag-drop, ask AI to swap things, customize. Shareable link. This is the web-first feature.

🧭 Companion Mode

Trip has started. App knows what day it is, what's next, where the hotel is. User sends natural language updates ("pizza tonight", "skip the museum, it's raining"). AI edits the live itinerary with context. This is where mobile matters most — design for it even in V1.

Key design decision: These two modes share the same itinerary data structure and the same 4 tools. The difference is the trip_context object that gets passed to Claude — in Companion Mode it includes today's date, current day number, last visited place, and user memory. In Planning Mode this object is empty.

Initial generation — no hotel

User types "Pondicherry, 3 days". AI generates itinerary using city center as the default anchor. Places are clustered geographically. Itinerary is saved and displayed.

User adds hotel

User types hotel name → lookup_place finds it → lat/lng stored in itineraries.hotel_lat/lng. UI shows a prompt: "We found your hotel. Want us to re-optimize each day's routing around it?"

trigger: hotel_added eventupdates: itinerary.hotel_place_id
3

Re-optimization pass

Claude is called with the existing itinerary + hotel coordinates. System prompt instructs: "Re-order each day so morning starts closest to hotel, and the day's final stop is within 1km of hotel. Minimize total walking distance." Only position/order changes — no places swapped unless Claude finds a clearly better nearby option.

operation: reorder_onlyshows: "saved 2.4km of walking on Day 2"

Map updates

Map re-renders with hotel pin (distinct icon), route lines updated per day. Each day's route now naturally starts and ends near hotel. User can still drag to override.

04

Real-Time Mood Updates

Live scenario · Day 2, 4:30pm, Pondicherry
"I'm in the mood for pizza tonight. Add a place that serves good Neapolitan."
01 → Claude receives message + trip_context: current day = 2, next stop = Serenity Beach, hotel = Villa Shanti (lat/lng known), time = 4:30pm
02 → Calls search_google_places with: type = restaurant, keyword = "Neapolitan pizza wood-fired", location = last stop lat/lng, radius = 2km. Returns ranked list with distances.
03 → Claude evaluates results. Finds two options: Option A — 5 min from Serenity Beach, rating 4.3. Option B — 20 min away, rating 4.7, frequently mentioned on Reddit r/Pondicherry.
04 → UI shows the trade-off explicitly — "Option A is 5 min away, Option B is 20 min but Redditors love it." User picks. Not hidden, not automatic. Respects user intelligence.
05 → Calls update_itinerary → adds dinner slot to Day 2 → map pin appears → preference likes_pizza written to user memory object.
05

The Four Tools

🔍
search_reddit
Pulls trip reports, hidden gems, and opinions from relevant subreddits. Cached 48hrs. The quality differentiator.
IN: destination, queries[], limit
OUT: posts[], comments[], upvotes
📍
lookup_place
Google Places lookup. Validates existence, gets lat/lng, hours, rating, price level, photo. Used for hotel lookup too.
IN: name, city, type?
OUT: lat, lng, hours, rating, place_id, photo_url
🗺️
search_nearby
NEW. Finds places near a given lat/lng. Used for real-time mood requests — "pizza near my last stop". Returns ranked by distance + rating.
IN: lat, lng, radius_km, keyword, type
OUT: places[] with distance, rating, open_now
✏️
update_itinerary
Creates, reorders, or patches an itinerary. Supports operations: generate, reorder, add_item, remove_item, swap_item. Triggers frontend re-render.
IN: itinerary_id, operation, payload
OUT: updated itinerary_id, changed days[]
Why 4 tools instead of 3: search_nearby is split from lookup_place because they serve different purposes. Lookup validates a known place. Nearby discovers unknown candidates. Claude needs both to handle mood requests intelligently — it searches nearby first, then lookups the winners to confirm details.

The Trip Context + Memory Object

Every AI call in Companion Mode includes this object. It's what makes the app feel like it knows you. Stored in Supabase, updated after each interaction.

trip_context.ts — passed with every Claude call during active trip
type TripContext = { // LIVE STATE — changes daily current_day: number // derived from trip start date trip_start_date: string // "2025-02-01" todays_date: string // injected at call time last_completed_stop: PlaceRef // { name, lat, lng } next_planned_stop: PlaceRef // ANCHORS — set once hotel: PlaceRef | null destination: string // "Pondicherry" // PREFERENCES — grow over trip food_preferences: { dietary: string[] // ["vegetarian"] cuisines_liked: string[] // ["neapolitan", "south indian"] cuisines_skipped: string[] // ["chinese"] price_range: string // "mid" | "budget" | "splurge" } activity_preferences: { pace: string // "relaxed" | "packed" liked_categories: string[] // ["beach", "heritage", "cafes"] skipped_places: string[] // place_ids user removed } // HISTORY — appended each interaction completed_stops: PlaceRef[] mood_updates: MoodUpdate[] // { time, message, action_taken } }

🧠 How preferences are learned

  • User removes a meat restaurant → dietary: ["vegetarian"] inferred
  • User adds pizza request → cuisines_liked: ["neapolitan"] added
  • User drags beach to morning every day → liked_categories: ["beach"], time preference noted
  • User skips a museum → skipped_places[], less museum suggestions
  • User consistently picks closer options → pace: "relaxed"

⚡ How it's used in the prompt

  • Injected into system prompt as JSON before each call
  • Claude told: "Use these preferences to filter and rank all suggestions"
  • Updated in Supabase after each tool call that changes preferences
  • Never shown raw to user — expressed as "based on your trip so far..."
  • Resets per trip, not per user (so next trip starts fresh)

trip_context as JSONB: Store the entire memory object as a single JSONB column on the itinerary row. Don't over-normalize this early. You'll iterate on what you track — JSONB lets you add fields without migrations. When you know exactly what you need, normalize the important bits.

Frontend Structure (React Native Ready)

project structure — shared hooks pattern
// Everything in /packages/core is shared between web and mobile packages/ ├── core/ ← Shared business logic (no JSX) │ ├── hooks/ │ │ ├── useItinerary.ts ← CRUD, reordering, optimistic updates │ │ ├── useTripContext.ts ← Live trip state, current day calc │ │ ├── useAIUpdate.ts ← Streaming AI calls, tool execution │ │ └── useMemory.ts ← Read/write trip_context preferences │ ├── api/ │ │ ├── supabase.ts ← Supabase client (same client, both platforms) │ │ └── claude.ts ← AI call wrapper with streaming │ └── types/ │ └── index.ts ← TripContext, Itinerary, ItineraryItem types │ apps/web/ ← Next.js app │ ├── app/page.tsx ← Landing — destination search │ ├── app/plan/[id]/page.tsx ← Split: Map (60%) + Panel (40%) │ │ ├── MapView.tsx ← Mapbox, pins by day color, hotel pin, routes │ │ ├── ItineraryPanel.tsx ← dnd-kit, day groups, place cards │ │ ├── HotelInput.tsx ← Search + confirm + trigger re-optimize │ │ └── MoodBar.tsx ← "What do you feel like?" input, bottom of panel │ └── app/api/ │ ├── generate/route.ts ← Streaming: full itinerary generation │ ├── update/route.ts ← Streaming: mood updates + edits │ └── tools/ │ ├── reddit.ts ← Reddit API + cache check/write │ ├── places.ts ← Google Places lookup │ └── nearby.ts ← Google Places nearby search │ apps/mobile/ ← React Native (future) ├── screens/PlanScreen.tsx ← Imports useItinerary from core ✓ └── screens/TripScreen.tsx ← Companion mode, imports useTripContext ✓
The monorepo pattern: Use a simple monorepo with packages/core for shared logic. Your hooks (useItinerary, useTripContext) work identically on web and mobile — only the UI components differ. When you start React Native, 60% of your logic is already written. Use pnpm workspaces — no need for Turborepo or Nx at solo scale.
